# 협업 방식 안내

이 문서는 프로젝트 시작 시점의 협업 규칙을 정리한 README 섹션입니다. 팀원들이 같은 규칙을 따르며 개발을 진행하도록 기준을 제공합니다.

---

## 1. 협업 방식 정하기

* 주간 코어타임(머지 상황에 따라) 동안 주요 동기화 및 머지 진행.
* 매일 또는 격일로 짧은 상태 공유 권장.
* 이슈 트래킹은 노션 사용.

---

## 2. 협업 프로세스

프로젝트는 이슈 기반으로 작업을 진행하며, 모든 변경 사항은 브랜치 → PR → developer → main 흐름으로 반영됩니다.

1. 이슈 생성
    - 진행할 작업의 이슈를 먼저 등록합니다.
    - 제목, 설명, 라벨을 명확하게 작성합니다.

2. 브랜치 생성 (developer 기준)
    - developer 브랜치를 기준으로 새로운 작업 브랜치를 생성합니다.
    - 브랜치 컨벤션 지켜서 생성합니다.

3. PR 생성
    - 작업이 완료되면 PR을 생성합니다.
    - PR의 base 브랜치는 항상 **developer** 입니다.

4. developer로 merge
    - 코드 리뷰가 끝난 PR은 developer 브랜치에 merge됩니다.
    - merge 후 개발 환경에서 정상 동작 여부를 확인합니다.

5. 확인 후 main으로 merge
    - developer에서 충분히 검증이 이루어지면 main 브랜치에 병합합니다.

---

## 전체 흐름 요약

1) 이슈 생성  
2) 브랜치 생성 (developer 기준)  
3) PR 생성 (base: developer)  
4) developer로 merge  
5) 확인 후 main으로 merge


---

## 3. 브랜치 컨벤션

브랜치 구조는 다음을 따릅니다:

자신의 이니셜 세글자로 하여 하이픈(`-`)을 붙인 후 다음 형식을 따릅니다.
(예 : 김민정 -> feature/kmj-<기능> )

* `main` — 항상 배포 가능한 상태 유지
* `dev` — 다음 배포를 위한 통합 브랜치(기능 병합 대상)
* `feature/<이니셜>-<기능>` — 새로운 기능 개발
* `fix/<이니셜>-<기능>` — `dev`에서 발생한 버그 수정용 브랜치

> 브랜치 이름은 소문자, 하이픈(`-`) 사용 권장. 의미를 알기 쉽게 짧고 구체적으로 작성.

---

## 4. 커밋 컨벤션

* 기본 형식: `[type] 내용`

* 자주 쓰는 `type` 예시:
  * `feat` — 새로운 기능
  * `fix` — 버그 수정
  * `refactor` — 코드 리팩터링
  * `docs` — 문서 수정
  * `chore` — 빌드/설정 등 사소한 변경
* 커밋 메시지는 목적이 명확하도록 짧고 현재형으로 작성합니다.

---

## 5. PR 컨벤션

* **기능 단위로 PR 생성**: 하나의 PR은 하나의 기능(또는 버그)을 중심으로 합니다.
* **PR 템플릿 적용**: PR에 목적, 변경사항 요약, 관련 이슈, 체크리스트(테스트·문서·리뷰 필요)를 기입합니다.
* **PR 제목 형식**: `[type] 내용`

  * 예: `[feat] 채팅방 생성 API 추가`, `[fix] 세마포어 초기화 race 조건 해결`

---

## 6. PR 단위 작업 흐름

1. 기능 개발 완료 → 개인 테스트/로컬 검증
2. `feature/*` 브랜치에서 `dev`로 PR 생성
3. 팀원 1명 이상이 코드를 리뷰
4. 리뷰 피드백 반영 후 다시 커밋
5. 최종적으로 코어타임에 맞춰 머지 진행

* **머지 방식**: 팀에서 합의한 방식(예: `Squash and merge` 또는 `Merge commit`)을 사용합니다. 자세한 규칙은 GitBook 또는 팀 문서 참고.
* PR은 기능 단위로 쪼개서 리뷰가 용이하도록 구성합니다.

---

## 7. 프로젝트 기한

* 세부 일정 및 마일스톤은 Git Book(또는 중앙 문서)에 따릅니다.
* Git Book에서 마감일과 각 스프린트 목표를 확인하세요.
