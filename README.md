# 협업 방식 안내

이 문서는 프로젝트 시작 시점의 협업 규칙을 정리한 README 섹션입니다. 팀원들이 같은 규칙을 따르며 개발을 진행하도록 기준을 제공합니다.

---

## 1. 협업 방식 정하기

* 주간 코어타임(머지 상황에 따라) 동안 주요 동기화 및 머지 진행.
* 매일 또는 격일로 짧은 상태 공유 권장.
* 이슈 트래킹은 노션 사용.

---

## 3. 브랜치 컨벤션

브랜치 구조는 다음을 따릅니다:

* `main` — 항상 배포 가능한 상태 유지
* `dev` — 다음 배포를 위한 통합 브랜치(기능 병합 대상)
* `feature/<이름>` — 새로운 기능 개발
* `fix/<이름>` — `dev`에서 발생한 버그 수정용 브랜치

> 브랜치 이름은 소문자, 하이픈(`-`) 사용 권장. 의미를 알기 쉽게 짧고 구체적으로 작성.

---

## 4. 커밋 컨벤션

* 기본 형식: `[type] 내용`

* 자주 쓰는 `type` 예시:
  * `feat` — 새로운 기능
  * `fix` — 버그 수정
  * `refactor` — 코드 리팩터링
  * `docs` — 문서 수정
  * `chore` — 빌드/설정 등 사소한 변경
* 커밋 메시지는 목적이 명확하도록 짧고 현재형으로 작성합니다.

---

## 5. PR 컨벤션

* **기능 단위로 PR 생성**: 하나의 PR은 하나의 기능(또는 버그)을 중심으로 합니다.
* **PR 템플릿 적용**: PR에 목적, 변경사항 요약, 관련 이슈, 체크리스트(테스트·문서·리뷰 필요)를 기입합니다.
* **PR 제목 형식**: `[type] 내용`

  * 예: `[feat] 채팅방 생성 API 추가`, `[fix] 세마포어 초기화 race 조건 해결`

---

## 6. PR 단위 작업 흐름

1. 기능 개발 완료 → 개인 테스트/로컬 검증
2. `feature/*` 브랜치에서 `dev`로 PR 생성
3. 팀원 1명 이상이 코드를 리뷰
4. 리뷰 피드백 반영 후 다시 커밋
5. 최종적으로 코어타임에 맞춰 머지 진행

* **머지 방식**: 팀에서 합의한 방식(예: `Squash and merge` 또는 `Merge commit`)을 사용합니다. 자세한 규칙은 GitBook 또는 팀 문서 참고.
* PR은 기능 단위로 쪼개서 리뷰가 용이하도록 구성합니다.

---

## 7. 프로젝트 기한

* 세부 일정 및 마일스톤은 Git Book(또는 중앙 문서)에 따릅니다.
* Git Book에서 마감일과 각 스프린트 목표를 확인하세요.

---

## 부록: 예시 템플릿

### 커밋 예시

```
[feat] 소셜 로그인 추가
[fix] 로그인 리다이렉트 오류 수정
[docs] README 협업 규칙 업데이트
```

### PR 템플릿 예시 (간단)

```
## 구현된 기능 요약
이번 PR에서 구현된 기능을 설명해주세요.

---

- 구현한 기능 목록
- 추가된 파일 또는 주요 로직
- 상세 변경 내용

---

- [ ] 유닛 테스트 통과

```
